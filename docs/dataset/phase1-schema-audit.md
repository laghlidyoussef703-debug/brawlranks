## 1. Schema inventory

**[REPO]** 25 migrations declare **45 tables**. `scripts/migrate.mjs` creates a 46th, `schema_migrations`, outside the migration files ŌĆö so a live database should show **46**, and a count of 45 means the bookkeeping table is missing.

**[REPO]** Structural facts, all generated by `schema-inventory.mjs`:

- Every table is `ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci`.
- Every primary key is a `CHAR(36)` UUID. There is no `AUTO_INCREMENT` anywhere.
- Every timestamp is `DATETIME(3)`. There are no `TIMESTAMP` columns, so no implicit timezone conversion exists.
- **16 LONGTEXT columns** are the high-growth surface. `raw_api_snapshots.payload` is the only one that is both `NOT NULL` and unbounded by design.
- **5 STORED generated columns** encode single-current-row invariants via the `IF(cond, 1, NULL)` pattern: `workflow_locks.active_flag`, `normalized_snapshots.accepted_flag`, `patches.active_flag`, `ranking_rule_sets.active_flag`, `published_snapshots.current_flag`.
- **36 CHECK constraints** enforce status/enum vocabularies.
- No routines, triggers, events, partitions, or views are declared anywhere.

### Schema drift risk

**[LIVE-REQUIRED]** The above is the *repository-declared* schema. It is not the live schema. Three drift scenarios are plausible and none can currently be excluded:

1. **Migration 0016 evidence of real drift.** **[REPO]** `migrations/0016_widen_workflow_runs_status.sql` exists precisely because a column width and its CHECK constraint disagreed in production. Drift of this class has already happened once in this project's history.
2. **Manual intervention.** A shared-hosting panel (phpMyAdmin) can alter a table without touching `schema_migrations`. The migration runner's checksum guard detects edited *files*, not edited *databases*.
3. **Partial migration application.** **[REPO]** `scripts/migrate.mjs` wraps each file in `beginTransaction()`, but MySQL and MariaDB both perform an implicit COMMIT on DDL. A file containing several DDL statements is therefore **not atomic**, and a mid-file failure leaves partial DDL applied with no `schema_migrations` row. Migrations 0009, 0018, 0023 and 0024 each contain multiple statements.

**Required action:** run sections 4, 5 and 6 of `production-size-report.sql` and diff against `schema-inventory.mjs --json`. Do not treat migrations as the live schema until that diff is clean.

### Tables that look purgeable but are not

DATASET.md warns against marking a table safe to delete merely because it is derived. Three cases where tracing changes the answer:

Deleting run metadata referenced by a published snapshot's ranking run breaks the provenance chain permanently. Child aggregate rows are rebuildable; the run row is not disposable.
- **`ranking_results` / `matchup_results` for held runs** are candidate data that never published. But **[CODE]** `lib/ranking/sync.ts:552` records `hold_reason = 'mass_movement_guard'` while retaining the full candidate set ŌĆö that set is the only evidence explaining *why* publication was withheld. **[PROD]** production is currently in exactly this state.
- **`normalized_players`** cannot be deleted while `battle_participants.player_id` references it **[REPO]** (`fk_battle_participants_player`, `migrations/0014`). It is ~214 MB and never independently prunable.

## 3. Highest-growth sources

**[PROD]** Reported largest objects: `matchup_aggregates` ~876 MB, `battle_participants` ~542 MB, `raw_api_snapshots` ~328 MB, `normalized_players` ~214 MB, `battle_teams` ~130 MB, `observed_players` ~130 MB, `normalized_battles` ~83 MB.

**[CODE]** The mechanism behind the largest object is the most important finding of this phase, and it is not raw battle growth:

`migrations/0022` makes every aggregate table's uniqueness include `aggregation_run_id`:

```sql
UNIQUE KEY uniq_matchup_aggregates_scope
  (brawler_id, opponent_brawler_id, game_mode_id, patch_id, aggregation_run_id)
```

**[CODE]** `lib/aggregation/sync.ts::executeNextAggregationSlice` creates three fresh `aggregation_runs` rows on every fresh start and writes a complete new aggregate set against them. There is no update path and no supersession. **[PROD]** one completed run produced 115,720 matchup aggregates; the measured 1,013,824 rows are Ōēł8.76 complete retained sets, at Ōēł100 MB per set.

**[CODE]** No retention query targets aggregates. `lib/ingestion/sync/retentionSweep.ts` sweeps exactly nine categories ŌĆö battle children, battles, raw snapshots, fetch runs, workflow steps, workflow runs, resolved incidents, unpromoted observed players, name history. `brawler_mode_aggregates`, `brawler_overall_aggregates`, `matchup_aggregates`, `ranking_results` and `matchup_results` appear in **none** of them.

**Conclusion:** the dominant growth source is unbounded accumulation of rebuildable aggregate sets, at roughly one new ~100 MB set per 6-hour scheduled invocation. That is ~400 MB/day of rebuildable data, which is why freezing aggregation/ranking (Phase 0, already done) was the correct emergency lever and why it is far safer than touching normalized history.

**[LIVE-REQUIRED]** The exact number of retained runs, and which are still referenced, needs sections 8 and 9 of `production-size-report.sql`.

## 4. Retention audit

**[CODE]** Deletion order in `lib/ingestion/sync/retentionSweep.ts` is FK-correct: battle children ŌåÆ battles ŌåÆ raw snapshots ŌåÆ fetch runs ŌåÆ workflow steps ŌåÆ workflow runs ŌåÆ incidents ŌåÆ observed players ŌåÆ name history. Batches are bounded at `RETENTION_BATCH_SIZE = 500` with `MAX_ITERATIONS_PER_CATEGORY = 20`, capping each category at 10,000 rows per invocation. `pruneFetchRunsOlderThan` and `pruneWorkflowRunsOlderThan` are `NOT EXISTS`-guarded against every referencing table. That part of the design is sound.

Four real risks:

### CRITICAL ŌĆö an empty request body performs a destructive sweep

**[CODE]** `app/api/internal/cron/retention-sweep/route.ts`:

```ts
let dryRun = false;
try {
  const body = await request.json().catch(() => null);
  if (body && typeof body.dryRun === "boolean") dryRun = body.dryRun;
} catch {
  // No body ŌĆö defaults to a real (non-dry-run) sweep.
}
```

**The question DATASET.md asks is answered: yes.** A POST with no body, an empty body, malformed JSON, `{}`, or `{"dryRun": "true"}` (a string, not a boolean) all leave `dryRun = false` and execute real deletions. The safe path requires an exactly-typed boolean; every failure mode falls through to destruction. The comment shows this is intentional, not a bug ŌĆö which makes it a design risk rather than a defect, but the blast radius is the same.

**[PROD]** Currently mitigated operationally only: the timer is disabled and `/opt/brawlranks-scheduler/run-job.sh` was changed to send `{"dryRun":true}`. Both mitigations live outside this repository. Anyone who re-enables the timer, restores the script backup, or calls the route manually with a valid `INTERNAL_CRON_SECRET` and no body triggers deletion.

**Proposed minimal fix (documented, NOT implemented):**

- Exact risk: unauthenticated-by-shape destructive default on an authenticated route.
- Affected path: `app/api/internal/cron/retention-sweep/route.ts` only.
- Minimal fix: invert the default to `dryRun = true` and require an explicit `{"dryRun": false}` to delete; return `400` on a non-boolean `dryRun`.
- Tests: extend `tests/phase4RoutesAuth.test.ts` with empty-body, `{}`, malformed-JSON, and string-`dryRun` cases asserting no deletion path is reached.
- Deployment impact: the scheduler payload already sends `{"dryRun":true}`, so behavior is unchanged until someone deliberately opts into deletion.
- Rollback: revert one file; no schema change, no data change.

**Not implemented here.** It is a production runtime behavior change requiring deployment, which is outside the Phase 1/2 boundary. DATASET.md assigns it to Phase 14 / WP6 ("archive-gated retention"). Implementing it now would also bundle a runtime change into a documentation-and-tooling commit.

### HIGH ŌĆö retention deletes normalized history with no archive

**[CODE]** `RETENTION_DAYS.NORMALIZED_BATTLE = 180` and `pruneNormalizedBattlesOlderThan` issues a plain `DELETE`. There is no archive, no manifest, no export. **[PROD]** an earlier `dryRun:false` run already deleted 12 `battle_children` and 12 `normalized_battles` rows. The data volume is trivial; the precedent is not ŌĆö it proves the destructive path executes against production. Under DATASET.md Phase 14 this must become archive-gated before it ever runs again.

### MEDIUM ŌĆö the sweep is not transactional across categories

**[CODE]** `sweepCategory` calls each prune function directly on the pool. Each `DELETE` is its own implicit transaction. A failure mid-sweep leaves earlier categories deleted and later ones untouched, with no manifest of what was removed. Rollback is impossible without a backup ŌĆö which is exactly why Phase 2 is the gate before any Phase 14 work.

### MEDIUM ŌĆö the battle-children prune can leave orphans on interruption

**[CODE]** `pruneBattleChildrenOlderThan` selects up to 500 battle ids, then issues three separate `DELETE`s (participants, teams, observations) outside a transaction, and `pruneNormalizedBattlesOlderThan` is a *separate* call. An interruption between them leaves battles whose children are gone. FK constraints do not prevent this, because deletion runs child-first. `validate-restored-db.sql` section 12 detects the resulting orphans.

## 5. Aggregation and ranking accumulation

Each DATASET.md question, answered from code:

| Question | Answer | Evidence |
|---|---|---|
| Does every run insert a complete new aggregate set? | **Yes.** Three new `aggregation_runs` rows per job, then full mode/overall/matchup passes over all active brawlers. | `lib/aggregation/sync.ts::executeNextAggregationSlice` |
| Are old runs retained indefinitely? | **Yes.** No retention path targets them. | `lib/ingestion/sync/retentionSweep.ts` (nine categories, none aggregate) |
| Are superseded aggregates referenced? | **Yes, transitively.** `ranking_runs` holds 3 FKs; `published_snapshots.ranking_run_id` reaches them. | `migrations/0024` |
| Can aggregation resume an incomplete run? | **Yes.** Cursor in `workflow_steps`; batch insert and cursor advance commit in one transaction. | `lib/aggregation/sync.ts` `withTransaction` |
| Are completed runs skipped or rebuilt? | **Rebuilt.** `findLatestRunningRun` only finds *running* runs; a completed one is invisible, so the next call starts fresh. | `lib/aggregation/sync.ts` |
| Do scheduler calls with `{}` always start new work? | **Yes** ŌĆö unless a run is already in progress, in which case they resume it. There is no freshness check and no minimum interval. | `app/api/internal/cron/aggregation-run/route.ts` ŌåÆ `stepAggregation` |
| Do ranking candidates accumulate when publication is held? | **Yes.** Full `ranking_results` + `matchup_results` sets are written in the `brawlers`/`matchups` phases, *before* the `publish` phase evaluates the guard. | `lib/ranking/sync.ts` phase order |
| Do `held_mass_movement` outcomes retain full candidate rows? | **Yes.** The hold only sets run status and skips snapshot creation; nothing is deleted. | `lib/ranking/sync.ts:552-559` |
| Are failed/abandoned runs cleaned up? | **No.** `reconcileStaleWorkflowRuns` marks a stale run failed after 15 min but deletes no rows. Partial aggregate rows from an abandoned job persist forever, unreferenced. | `lib/workflow.ts`, `lib/aggregation/sync.ts` |
| Can runs overlap or race? | **No.** `workflow_locks` unique `(workflow_definition_id, active_flag)` serializes per workflow; a second caller gets `lock_not_acquired` / HTTP 409. | `lib/workflow.ts::acquireWorkflowLock`, `migrations/0002` |
| Do published snapshots depend on historical aggregate run rows? | **Yes**, via `published_snapshots ŌåÆ ranking_runs ŌåÆ aggregation_runs`. Public *reads* do not, but provenance does. | `migrations/0024` |
| What can be archived without breaking rollback or reads? | Child aggregate detail and candidate rows for completed runs that are not current, not held, and not transitively referenced by any published snapshot. Run metadata must stay. | synthesis of the above |

**Abandoned-run garbage is a distinct, unmeasured category.** **[LIVE-REQUIRED]** Section 8 of `production-size-report.sql` joined against section 9 will reveal aggregate rows whose run is `running`/`failed` and unreferenced ŌĆö orphaned work with no consumer at all.

**[PROD]** Ranking has repeatedly evaluated 106 brawlers and returned `held_mass_movement` with `brawlersPublished: 0` and `tierMoveRatio` Ōēł0.56ŌĆō0.65. **[CODE]** `lib/ranking/formulas.ts:214` holds when the ratio exceeds `0.25`. A ratio above 0.5 means more than half the brawlers changed tier between runs ŌĆö consistent with a young, rapidly-growing dataset where percentile tiers are still unstable. **This is the guard working as designed, not a fault**, and it must not be mistaken for a migration failure later (DATASET.md Phase 11 makes the same point). Each held run still costs a full candidate set.
