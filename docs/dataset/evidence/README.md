### Server semantics (restored MariaDB, for the MySQL 8.4 gate)

```
VERSION()              11.8.8-MariaDB-ubu2404
@@sql_mode             STRICT_TRANS_TABLES,ERROR_FOR_DIVISION_BY_ZERO,
                       NO_AUTO_CREATE_USER,NO_ENGINE_SUBSTITUTION
@@time_zone            SYSTEM
@@character_set_server utf8mb4
@@collation_server     utf8mb4_uca1400_ai_ci   <-- MariaDB-only server default
```

Two compatibility notes fall directly out of this evidence, both already
covered by `scripts/dataset/compatibility-check.mjs` and Phase 3 tooling:

1. **Server default collation** is `utf8mb4_uca1400_ai_ci` (a MariaDB-only UCA
   collation). It is **not** used by any table — every table explicitly declares
   `utf8mb4_unicode_ci`, which exists on MySQL 8.4 — so the schema is portable.
   The MySQL 8.4 target must still be created with an explicit
   `utf8mb4_unicode_ci` default rather than MySQL's `utf8mb4_0900_ai_ci`.
2. **`@@sql_mode` does not include `ONLY_FULL_GROUP_BY`** (MariaDB default),
   whereas MySQL 8.4 enables it by default. This is the sql_mode diff Phase 3
   must reconcile in staging.

`battle_teams.rank` is confirmed `int(11) NULL` on the live copy — the reserved
word that is the MySQL 8.4 migration blocker (see
`docs/dataset/phase3-mysql84-compat.md`).
